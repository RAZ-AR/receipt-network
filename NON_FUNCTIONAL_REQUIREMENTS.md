# Non-Functional Requirements

## Производительность

- Экран должен подтверждать загрузку чека практически сразу.
- Результат запроса к TaxCore/PURS verification API (основной путь) или предварительный OCR-результат (fallback) должен появляться быстро.
- Длительная проверка должна выполняться асинхронно с понятным статусом.
- При недоступности TaxCore/PURS verification API система должна переключаться на понятный статус ожидания и повторные попытки, не блокируя пользователя навсегда.

## Надежность

- Начисление баллов не должно дублироваться.
- Создание билета должно быть идемпотентным.
- Каждое изменение баланса должно иметь журнал.
- Критические операции должны быть повторяемыми после сбоя.

## Безопасность

- Шифрование данных при передаче и хранении.
- Разделение ролей.
- MFA для администраторов.
- Ограничение доступа сотрудников бизнеса.
- Защита от массовых запросов.
- Аудит чувствительных действий.

## Приватность

- Минимизация персональных данных.
- Отдельные согласия на маркетинг.
- Удаление аккаунта и данных.
- Геолокация только по согласию.
- Бизнес получает только разрешенные данные.

## Масштабируемость

- OCR и обработка чеков должны масштабироваться независимо.
- Модули должны иметь четкие доменные границы.
- Возможность позднее выделить тяжелые компоненты в отдельные сервисы.

## Наблюдаемость

- логи;
- метрики;
- трассировка;
- алерты;
- fraud monitoring;
- аудит операций;
- dashboard по обработке чеков.

## Локализация

MVP должен учитывать:

- сербский язык;
- латиницу;
- кириллицу;
- локальные форматы дат;
- RSD;
- локальные фискальные реквизиты.
